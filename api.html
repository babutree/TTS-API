Authentication

The service has built-in API Key authentication so external programmatic clients (browser extensions, scripts, other backends) can call it directly. Rules:

  • Where the key comes from: set the TTS_API_KEY environment variable in docker-compose.yml. Change it to your own strong random value.
  • Empty (unset) = fully open: any request is allowed, keeping local direct-connect usage working.
  • Same-origin pages are exempt: this docs page and the main UI (/index.html) are same-origin with the backend and need no key in a browser.
  • External clients must send the key: REST uses header X-API-Key: <key> (preferred, especially behind Caddy Basic Auth) or Authorization: Bearer <key>; WebSocket cannot set custom headers in a browser, so use query parameter /ws/tts?key=<key>.

Honest boundary: same-origin exemption relies on the Origin/Referer header, which non-browser clients can forge. For real network isolation still put a reverse proxy (Caddy) in front; the key provides controlled access for external integrations.

# CRX plugin call example (external client, key required)
fetch("https://tts.example.com/api/tts", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    "X-API-Key": "YOUR_SECRET_KEY"
  },
  body: JSON.stringify({ text: "Hello", engine: "kokoro", voice: "zf_xiaoxiao", speed: 1.0 })
});

GET/

Health check. Returns the engine and ffmpeg readiness status.

Response 200 — engine ready

{ "status": "v0.1 engine running", "ready": true }

Response 503 — engine starting

{ "status": "starting", "ready": false }

Response 503 — ffmpeg missing

{ "status": "ffmpeg missing", "ready": false }

GET/api/auth

Key-probe endpoint: verifies whether the supplied key is valid, so a client (e.g. a CRX extension) can check credentials before real calls. This endpoint does not honor same-origin exemption — even the bundled pages must send a correct key to get authorized: true.

Response 200 — authorized (or auth disabled)

{ "auth": "enabled", "authorized": true }
{ "auth": "disabled", "authorized": true }   // when TTS_API_KEY is not configured

Response 401 — key missing or wrong

{ "auth": "enabled", "authorized": false, "detail": "缺少或错误的 API Key" }

Use the "Verify key" button in the Authentication section above to test this endpoint.

GET/api/logs

Diagnostic log endpoint for the in-memory ring buffer. It is not same-origin exempt; when TTS_API_KEY is configured, send a valid key with X-API-Key or Authorization: Bearer.

{ "limit": 100, "total_buffered": 287, "lines": ["2026-... INFO tts-api: ..."] }

GET/api/voices/preview

Synthesize a short built-in voice preview. Query parameters: engine (kokoro or edge), voice, and speed (0.5–3.0). The response is audio/mpeg with Content-Disposition: inline.

WS/ws/tts

Interactive TTS session. Send a JSON request, receive a stream of binary PCM frames interleaved with JSON control messages. The connection stays open after a request completes, so you can send the next request.

Client → Server

{
  "text": "第一句。\n第二句。",
  "engine": "kokoro",
  "voice": "zf_xiaoxiao",
  "speed": 1.0
}

Separate sentences with \n — the server uses these as synthesis units and aligns segment boundaries 1:1 for gapless streaming. speed is clamped server-side to 0.5–3.0.

Server → Client

MessageTypeDescription
{"type":"start"}JSONBegin processing a request
{"type":"seg","text":"..."}JSONSentence boundary marker; marks the start of one synthesis unit in the PCM stream
Binary frameArrayBuffer24kHz 16-bit mono PCM chunk (typically 2048 bytes)
{"type":"end"}JSONAll audio sent; connection stays open
{"type":"error","message":"..."}JSONRequest rejected or synthesis failed; connection stays open

error covers: invalid JSON, missing/empty text, text over length, unknown engine, unknown Kokoro voice, and synthesis failure (e.g. an invalid Edge voice or engine fault). On synthesis failure the server sends error instead of end, so the client leaves the "synthesizing" state without a false success.

Cancel / Interrupt

While the server is synthesizing, the client can send any message (or close the connection) to cancel the current request. The server immediately stops synthesis and starts listening for the next request.

Auto engine

"engine":"auto" is a client-side UI option only — it is never sent to the server. When Auto is selected, the browser splits by sentence and language boundary: Chinese (including Chinese punctuation) routes to the Chinese voice, and English (including single terms/all-caps abbreviations like DNS, OpenWrt) always routes to the English voice. Each Kokoro pipeline is monolingual — the Chinese pipeline cannot even pronounce isolated English abbreviations (official issues #95/#238), so English is never merged into a Chinese segment.

Text preprocessing

Before synthesis the server strips Markdown formatting (code blocks, inline code, images, links, headings, blockquotes, lists, bold/italic/strikethrough, horizontal rules) and removes quotation marks (straight/curly double quotes, CJK corner brackets, guillemets), preventing markup from being read aloud or quotes producing phoneme artifacts in Kokoro.

Additionally Kokoro applies per-voice language filtering (a fallback safeguard): Chinese voices (zf_*/zm_*) strip all English letters before synthesis, English voices (af_*/am_*) strip all CJK characters. Each Kokoro pipeline is monolingual and mispronounces cross-language text, so stripping the other language is preferable to reading garbage. For mixed Chinese/English text, use Auto mode in the UI to route per sentence.
